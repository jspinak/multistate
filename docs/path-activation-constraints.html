Path Activation Constraints

Overview​

When searching for paths with MultiState, the framework finds paths that activate all specified target states, but currently does not restrict the path from activating additional states beyond those targets.

Current Behavior: Inclusive Activation​

The formal model defines path validity as:

valid(ρMT)  ⟺  Stargets⊆⋃i=0n{si}\text{valid}(\rho_{MT}) \iff S_{targets} \subseteq \bigcup_{i=0}^{n} \{s_i\}valid(ρMT​)⟺Stargets​⊆⋃i=0n​{si​}

This means:

  • ✅ The path must visit ALL target states
  • ✅ Additional states may be activated along the path
  • ✅ Any configuration that includes the target states is acceptable

Example​

If you specify targets [StateA, StateB], the pathfinder might return a path that activates:

  • StateA → StateB → StateC (valid: includes both targets plus StateC)
  • StateD → StateA → StateB → StateE (valid: includes both targets plus StateD and StateE)

Exclusive Activation (Not Currently Supported)​

An "exclusive activation" mode would require that ONLY the specified target states end up active, with no additional states. This would require a stricter constraint:

Sfinal=StargetsS_{final} = S_{targets}Sfinal​=Stargets​

Where S_final is the final set of active states after the path executes.

Use Cases for Exclusive Activation​

Exclusive activation would be useful for:

  • Strict state configurations: When you need exactly a specific set of states active
  • Resource management: When additional states consume resources you want to avoid
  • Testing scenarios: When validating that only intended states are reachable
  • Security contexts: When additional states might expose unintended functionality

Current Workarounds​

While exclusive activation isn't directly supported, you can:

  1. Post-path validation: Check the final active states after path execution
  2. Custom path filtering: Implement additional logic to filter paths based on their complete state activation
  3. Use blocking states: Configure states to block undesired state activations
  4. Design state groups: Use atomic state groups to control which states activate together

Implementation Considerations​

Adding exclusive activation support would require:

  1. New path constraint type: Distinguish between inclusive and exclusive target specifications
  2. Modified pathfinding algorithm: Prune paths that activate non-target states
  3. Performance implications: Exclusive constraints could significantly reduce the solution space, potentially making some paths impossible to find

Summary​

  • Current: Paths must include all targets, may activate additional states
  • Future consideration: Exclusive mode where ONLY targets are activated
  • Workaround: Post-validation or custom filtering of generated paths
